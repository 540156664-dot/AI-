# AGV 工控机硬件监控程序 — 需求文档

> 版本: v1.1 | 日期: 2026-05-20 | 目标平台: Ubuntu 18.04 + ROS1 | CAN 硬件: 宜鼎 Innodisk

---

## 1. 项目背景

### 1.1 现状与痛点

AGV 工控机运行 ROS1 系统，CAN 报文及各传感器信号已通过 ROS 节点采集并以 MSG 形式发布。但在长期无人值守运行时，存在以下问题：

- **CAN 总线掉线无感知**：CAN 总线因线缆松动、终端电阻失效、收发器故障等原因中断后，ROS 节点可能仍在发布过时数据，主控程序无法区分
- **系统资源不可见**：工控机 CPU 过热降频、内存泄漏导致的 OOM、磁盘写满导致日志丢失，这些问题在事发前无任何告警
- **故障定位困难**：现场人员无法判断是 CAN 硬件故障、网络中断、还是系统资源耗尽

本程序作为 ROS1 节点运行，订阅已有 Topic 获取信号数据，同时采集系统资源（CPU/内存/磁盘），提供终端监控界面。

### 1.2 与现有 CAN 诊断工具的区别

| 维度 | 现有 CAN 诊断工具（Windows） | 本程序（Ubuntu 工控机） |
|------|------------------------------|---------------------------|
| 运行环境 | Windows 10/11 调试笔记本 | Ubuntu 18.04 车载工控机 |
| CAN 硬件 | — | 宜鼎 Innodisk (SocketCAN/mPCIe) |
| 用户 | 调试工程师 | 车载主控程序 + 现场运维人员 |
| 数据来源 | USB-CAN 设备直读 | ROS1 Topic 订阅（复用已有采集节点） |
| 核心目标 | 协议分析、信号可视化 | 健康监测、故障告警 |
| 通道数 | 单路 CAN | 2~4 路 CAN（工控机多路 CAN 接口） |
| 配置 | DBC / 自定义格式 | 复用 DeviceCanModel / VehicleModel JSON 配置 |

## 3. 功能需求

### F1 — 系统资源监控

实时采集并展示工控机硬件资源使用情况。**监控采集默认关闭，通过手动按钮触发启动/停止，避免长期运行占用系统资源**。

| 监控项 | 采集方式 | 指标 | 告警阈值(默认) |
|--------|----------|------|----------------|
| CPU 使用率 | `/proc/stat` 或 `psutil` | 总体使用率/每核使用率 | > 80% 黄色告警, > 95% 红色告警 |
| CPU 温度 | `/sys/class/thermal/` 或 `sensors` | 核心温度(℃) | > 80℃ 黄色, > 90℃ 红色 |
| 内存使用 | `/proc/meminfo` 或 `psutil` | 总量/已用/可用/百分比 | > 80% 黄色, > 95% 红色 |
| 磁盘使用 | `os.statvfs` 或 `psutil` | 分区总量/已用/可用/百分比 | > 80% 黄色, > 95% 红色 |
| 网络状态 | `/sys/class/net/` | 各网卡 UP/DOWN、丢包率 | 网卡 DOWN 即告警 |
| 系统负载 | `os.getloadavg` | 1min/5min/15min 平均负载 | > CPU 核心数 × 1.5 |
| 进程存活 | PID 文件或进程名检测 | AGV 主控进程是否运行 | 进程退出即告警 |

#### F1.1 历史回放

支持对已记录的硬件监控数据进行回放查看：

| 回放项 | 说明 |
|--------|------|
| 回放数据源 | 从本地日志文件加载历史数据 |
| 回放控制 | 播放/暂停/停止、倍速切换（1×/2×/5×/10×） |
| 时间轴拖拽 | 支持拖拽时间轴跳转到指定时刻 |
| 回放范围 | CPU/内存/磁盘/温度/网络等全部系统指标 |
| 显示方式 | 趋势折线图 + 数值表格，与实时采集界面一致 |

### F2 — CAN 总线掉线诊断

**核心功能**：通过 ROS1 Topic 订阅获取 CAN 总线及节点状态数据，区分不同程度的问题并给出清晰告警。**监控默认关闭，手动开启，避免长期占用系统资源**。

#### F2.1 CAN 硬件状态检测

CAN 接口物理状态仍需直接从系统检测（ROS 无法反映硬件层故障）：

| 检测项 | 说明 | 检测方式 |
|--------|------|----------|
| CAN 接口是否存在 | `/sys/class/net/can0` 是否存在 | 检测网络接口列表 |
| CAN 接口是否 UP | `ip link show can0` 的 state | 读取接口 flags |
| CAN 控制器错误 | 收发器故障、总线短路等 | 读取 CAN 控制器错误计数器 (`ip -details -statistics link show can0`) |
| 总线负载率 | 当前总线占用率 | 统计单位时间内报文数及位数 / 波特率 |

#### F2.2 节点心跳监测

通过 ROS1 Topic 订阅检测各节点在线状态，**订阅节点列表由配置文件管理**：

| 检测项 | 说明 |
|--------|------|
| 节点在线状态 | 通过 Topic 超时判断节点是否在线 |
| 心跳超时时间 | 可配置，默认 500ms |
| 离线告警 | 节点超时未收到 Topic 数据 → 红色告警，记录离线时刻和持续时间 |
| Topic 配置 | 通过配置文件管理订阅哪些 ROS Topic 及对应节点 |

#### F2.3 CAN 掉线诊断流程

```
┌──────────────┐
│ 定时检测循环  │ (每 1 秒)
└──────┬───────┘
       ▼
┌──────────────┐    异常    ┌─────────────────┐
│ can0 接口存在?│ ────────► │ 告警: CAN硬件未检测到│
└──────┬───────┘           └─────────────────┘
       │正常
       ▼
┌──────────────┐    异常    ┌─────────────────┐
│ can0 接口 UP? │ ────────► │ 告警: CAN接口DOWN  │
└──────┬───────┘           └─────────────────┘
       │正常
       ▼
┌──────────────┐    异常    ┌─────────────────┐
│ TEC/REC < 阈值?│ ────────► │ 告警: 错误计数过高  │
└──────┬───────┘           └─────────────────┘
       │正常
       ▼
┌──────────────┐    超时    ┌─────────────────┐
│ ROS Topic 有心跳?│ ──────► │ 告警: XX节点离线   │
└──────┬───────┘           └─────────────────┘
       │正常
       ▼
  🟢 系统正常
```

### F3 — 信号数据解析

**数据来源**：ROS1 系统已将 CAN/传感器信号采集并发布为 Topic，本程序直接订阅消费，不直接读取 SocketCAN。

#### F3.1 ROS Topic 订阅

- 订阅 ROS1 已有 Topic 获取 CAN 报文及传感器数据
- 订阅的 Topic 列表通过配置文件管理，可切换显示不同节点数据
- 环形缓冲区存储最近 N 条数据（默认 10000 条）

#### F3.2 配置复用

复用现有 CAN 诊断工具的 JSON 配置文件进行信号解析：

| 配置文件 | 内容 |
|----------|------|
| `DeviceCanModelObjectConfig.json` | 设备模型定义：CAN-ID、信号位偏移(`tagOffsetAddress`)、数据类型(`tagDataType`)、协议类型、名称描述 |
| `VehicleModelObjectConfig.json` | 整车配置：各设备挂载的 CAN 路数(`canNumber`)、节点 ID(`canNodeID`)、设备参数标定 |

支持的解析能力：
- 按 CAN-ID → 设备/节点匹配
- 按字节偏移 + 位偏移 + 长度解析信号
- 支持数据类型：uint8/uint16/uint32/int8/int16/int32/float/bit
- 支持字节序：Intel LSB / Motorola MSB
- 缩放系数 + 偏移量
- CANopen 协议子集：EMCY 紧急报文、Heartbeat、NMT 状态

#### F3.3 解析结果输出

- 结构化 JSON 输出，供终端界面消费
- 人类可读日志格式

### F6 — 数据记录

**数据记录默认关闭，手动按钮触发开启/停止**，避免持续写盘占用系统资源。

- **系统指标记录**：手动开启后，每 5 秒记录一次 CPU/内存/磁盘值到 CSV 文件
- **CAN 报文记录**：手动开启后，记录所有 CAN 报文到文件（与现有工具的录制功能类似）
- **告警事件记录**：所有告警的触发/恢复时间线（告警检测始终运行，不受按钮控制）
- **日志轮转**：自动清理超过 7 天的日志文件
- **存储位置**：`~/agv-monitor/logs/`

## 4. 技术方案

### 4.1 技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| 语言 | Python 3.6+ (Ubuntu 18.04 自带) | 与 ROS1 rospy 无缝集成 |
| 数据来源 | ROS1 Topic 订阅 (`rospy`) | CAN/传感器信号已由 ROS 节点采集发布，直接订阅消费 |
| CAN 硬件检测 | SocketCAN (`/sys/class/net/`) | 硬件层故障检测需直接读内核接口 |
| 终端界面 | Python `curses` / 直接终端输出 | 无需浏览器，SSH 即可查看 |
| 系统监控 | `psutil` + `/proc` | CPU/内存/磁盘/温度采集 |
| 配置格式 | JSON | 复用现有 DeviceCanModel / VehicleModel 配置，与现有工具统一 |
| 进程管理 | ROS 节点启动 | 作为 ROS 节点运行，随 ROS 系统启动 |

### 4.2 关键依赖

```
psutil>=5.0
rospy  (ROS1 自带，用于 Topic 订阅)
```

ubuntu 18.04 安装:
```bash
sudo apt install python3-pip can-utils ros-melodic-desktop-full
sudo pip3 install psutil
```

### 4.3 Innodisk CAN 驱动说明

宜鼎 (Innodisk) CAN 模块在 Ubuntu 18.04 下的驱动方式：

- **型号参考**: EMUC-B202 (mPCIe CAN 卡), EMUC-B201 等
- **驱动**: 宜鼎提供 Linux SocketCAN 驱动，编译后生成 `.ko` 内核模块
- **接口**: 加载驱动后，CAN 接口以 `can0`/`can1` 等形式出现在 `/sys/class/net/` 下
- **硬件检测**: 通过 `/sys/class/net/` 和 `ip link` 读取接口状态和错误计数器
- **ROS 层**: 已有 ROS 节点通过 SocketCAN 采集 CAN 报文并发布为 Topic，本程序不直接操作 SocketCAN 收发
