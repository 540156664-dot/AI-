# 项目文档管理系统 — 需求与设计文档

> 版本: v1.0 | 日期: 2026-05-20 | 作者: 项目管理工程师

---

## 一、项目背景

项目管理过程中，文件分散在不同目录，命名混乱、版本不清、查找困难。需要一个轻量级本地工具，统一管理、归档、检索所有项目文档。

---

## 二、技术方案

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端 | Python 3 + FastAPI + uvicorn | 轻量 Web 框架，文件处理能力强 |
| 数据库 | SQLite + SQLAlchemy | 零配置，元数据存储 |
| 前端 | HTML/CSS + Alpine.js + htmx | 无构建步骤，现代交互体验 |
| 文件存储 | 本地磁盘 UUID 命名 | 文件统一存入 `storage/` 目录 |
| 启动 | 单脚本 `python app.py` | 浏览器访问 `http://localhost:8520` |

## 三、功能需求

### 3.1 文件入库（混合模式）

**方式一：手动上传**
- 页面提供拖拽区域，支持拖入单个或多个文件
- 点击拖拽区域可弹出文件选择框
- 支持拖入整个文件夹（递归读取）

**方式二：目录扫描导入**
- 用户可配置 1~N 个"监控目录"
- 支持手动触发扫描，列出目录中尚未入库的新文件
- 用户勾选确认后批量导入
- 支持"仅扫描"或"扫描后移动文件"两种模式

**入库行为：**
- 文件以 UUID 重命名，存入 `storage/` 统一仓库目录
- 原始文件名、原始路径、文件大小、MD5 哈希、入库时间均记录在数据库
- 默认"复制"模式（不删原文件），可选"移动"模式（入库后删除原文件）

**版本管理：**
- 同一文件（MD5 相同）自动去重，不会重复存储
- 允许覆盖更新：上传同名文件时，可选择"创建新版本"或"替换"

### 3.2 标签系统

**系统预置标签：**

| 标签类别 | 可选值 |
|----------|--------|
| 项目名称 | （用户配置，如"XX大厦"/"YY公路"/"ZZ改造"） |
| 文档类型 | 合同 / 图纸 / 报告 / 会议纪要 / 签证单 / 验收单 / 联系单 / 其他 |
| 项目阶段 | 启动 / 规划 / 执行 / 监控 / 收尾 / 归档 |
| 年份 | （自动提取，也可手动覆盖） |

**自定义标签：**
- 用户可创建任意标签（名称 + 颜色）
- 标签支持分组管理
- 可为单个文件打多个标签
- 支持批量打标签（选中多个文件，统一添加标签）

### 3.3 文件命名规范校验

**目的：** 防止文件随意命名，确保归档文件命名统一可读。

**机制：** 为每种"文档类型"标签配置一个命名模板。文件入库时，系统自动校验文件名是否匹配模板，不匹配则拒绝入库并提示正确格式。

**命名模板：**
- 模板由固定文字 + 占位符组成，占位符用 `[ ]` 标记
- 每个占位符代表一个必填字段，系统拆分验证每段是否存在且非空
- 占位符之间用下划线 `_` 或中划线 `-` 连接

**模板示例：**

| 文档类型 | 命名模板 |
|----------|----------|
| 合同 | `[项目名]_[合同类型]_[签订日期]_[对方单位]` |
| 技术协议 | `[项目名]_技术协议_[签订日期]_[对方单位]` |
| 图纸 | `[项目名]_[图纸专业]_[图号]_[版本]` |
| 签证单 | `[项目名]_签证单_[编号]_[日期]` |
| 会议纪要 | `[项目名]_会议纪要_[日期]_[会议主题]` |
| 验收单 | `[项目名]_验收单_[验收内容]_[日期]` |

**校验流程：**
1. 用户上传文件 → 选择文档类型标签
2. 系统查找该文档类型是否配置了命名模板
3. 有模板 → 校验文件名，不符合则弹窗提示正确格式，**拒绝入库**
4. 无模板 → 不校验，直接通过

**示例交互：**
```
上传文件: "施工合同最终版.pdf"
文档类型: 合同
命名模板: [项目名]_[合同类型]_[签订日期]_[对方单位]

❌ 文件名不符合命名规范！
  您的文件名: 施工合同最终版.pdf
  要求格式:  [项目名]_[合同类型]_[签订日期]_[对方单位]
  正确示例:   xx大厦_施工总包合同_20260501_xx建设公司.pdf
```

**模板管理：**
- 标签管理页面中，文档类型标签可编辑其命名模板
- 模板中的占位符名可自定义（如 `[对方单位]` 可改为 `[供应商]`）
- 预览区域实时展示"用户看到的要求格式"和"一个正确示例"

### 3.4 搜索与筛选

**搜索方式：**
- **关键词搜索**：匹配原始文件名（模糊搜索）
- **标签筛选**：多标签组合，支持 AND（同时满足）和 OR（满足其一）
- **日期范围**：按入库时间或文件修改时间筛选
- **文件类型**：按扩展名过滤（pdf/docx/xlsx/dwg/jpg 等）
- **组合筛选**：以上条件可任意组合

**全文搜索（二期可选）：**
- 对 `.txt` `.md` `.pdf` `.docx` `.xlsx` 文件建立文本索引
- 支持搜索文件内容

### 3.5 浏览与预览

**文件列表视图：**
- 网格视图（缩略图卡片）和列表视图（表格）可切换
- 默认按入库时间倒序
- 支持按名称、大小、类型、日期排序
- 分页或无限滚动

**文件详情面板：**
- 点击文件展示：原始文件名、UUID 存储名、文件大小、MD5、入库时间、已有标签
- 可在此面板添加/移除标签

**预览支持：**
- 图片（jpg/png/gif/svg/webp）：直接显示
- PDF：浏览器内嵌预览
- 文本文件（txt/md/log/csv）：代码高亮显示
- Office 文档：提供下载按钮（暂不支持在线预览）
- 其他类型：显示文件信息 + 下载按钮

**文件操作：**
- 下载文件（自动恢复原始文件名）
- 删除文件（从存储和数据库中移除，需确认）
- 替换文件（上传新版本，保留标签和元数据）

### 3.6 仪表盘

- 文件总数 / 标签总数 / 项目总数
- 最近入库的文件列表（最近 10 条）
- 按文档类型的分布统计（饼图或柱状图）
- 按项目维度的文件数量统计
- **待处理的核心文件缺失提醒**（醒目展示）

### 3.7 核心文件检查（重点功能）

**业务场景：** 项目关键节点（如进厂前、开工前、验收前）必须确保所有必需文件已归档，缺少任何一项都是合规风险。

**检查清单模板：**
- 用户可创建多个检查清单模板，每个模板对应一个项目节点
- 例如："进厂前必查清单"包含：施工合同、技术协议、能源供应协议、安全协议、进场许可证、人员资质证明……
- 每个清单项绑定一个"文档类型"标签（如"合同"/"技术协议"/"资质证明"），并可附加备注说明

**执行检查：**
- 选择"项目 + 检查清单" → 系统自动逐项检查该项目下是否存在对应文档类型标签的文件
- 已就绪项显示绿色 + "已归档（N 份）"
- 缺失项显示红色 + "⚠ 缺失"
- 结果可导出为检查报告（打印/PDF/复制文本）

**检查结果汇总：**
- 一目了然的通过/未通过状态
- 缺失项优先置顶展示
- 如全部通过，显示"✅ 所有核心文件已就绪"

**检查清单示例：**

| 序号 | 必需文件 | 文档类型标签 | 状态 |
|------|----------|-------------|------|
| 1 | 施工合同 | 合同 | ✅ 已归档（3份） |
| 2 | 技术协议 | 技术协议 | ✅ 已归档（1份） |
| 3 | 能源供应协议 | 能源协议 | ⚠ 缺失 |
| 4 | 安全施工协议 | 安全协议 | ✅ 已归档（1份） |
| 5 | 进场许可证 | 许可证 | ⚠ 缺失 |
| 6 | 人员资质证明 | 资质证明 | ✅ 已归档（5份） |

## 四、数据库设计

```sql
-- 文件表
CREATE TABLE files (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    uuid_name     TEXT NOT NULL UNIQUE,          -- 存储用的 UUID 文件名(含扩展名)
    original_name TEXT NOT NULL,                 -- 原始文件名
    original_path TEXT,                          -- 原始路径（扫描导入时记录）
    file_size     INTEGER NOT NULL,              -- 字节
    file_type     TEXT,                          -- 扩展名，如 pdf
    md5_hash      TEXT NOT NULL,                 -- MD5 哈希，用于去重
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 标签表
CREATE TABLE tags (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    name            TEXT NOT NULL UNIQUE,               -- 标签名称
    category        TEXT,                               -- 标签类别（项目/类型/阶段/自定义）
    color           TEXT DEFAULT '#3B82F6',             -- 标签颜色
    naming_template TEXT                                -- 命名模板（仅文档类型标签使用）
);

-- 文件-标签关联表
CREATE TABLE file_tags (
    file_id INTEGER NOT NULL REFERENCES files(id) ON DELETE CASCADE,
    tag_id  INTEGER NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (file_id, tag_id)
);

-- 扫描目录配置表
CREATE TABLE watch_dirs (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    path      TEXT NOT NULL,
    enabled   INTEGER DEFAULT 1
);

-- 检查清单模板表
CREATE TABLE checklists (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL,                    -- 清单名称，如"进厂前必查清单"
    description TEXT,                             -- 清单说明
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 检查清单明细表
CREATE TABLE checklist_items (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    checklist_id INTEGER NOT NULL REFERENCES checklists(id) ON DELETE CASCADE,
    sort_order   INTEGER DEFAULT 0,               -- 排序序号
    item_name    TEXT NOT NULL,                    -- 检查项名称，如"施工合同"
    tag_name     TEXT NOT NULL,                    -- 对应的文档类型标签名，如"合同"
    description  TEXT,                             -- 补充说明
    required     INTEGER DEFAULT 1                 -- 是否必须（1=必须, 0=可选）
);

## 五、API 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/files` | 文件列表（支持筛选、分页、排序参数） |
| POST | `/api/files/upload` | 上传一个或多个文件 |
| GET | `/api/files/{id}` | 文件详情 |
| GET | `/api/files/{id}/download` | 下载文件（恢复原始文件名） |
| DELETE | `/api/files/{id}` | 删除文件 |
| POST | `/api/files/{id}/tags` | 为文件添加标签 |
| DELETE | `/api/files/{id}/tags/{tag_id}` | 移除文件标签 |
| GET | `/api/tags` | 获取所有标签 |
| POST | `/api/tags` | 创建标签 |
| DELETE | `/api/tags/{id}` | 删除标签 |
| PUT | `/api/tags/{id}` | 修改标签（名称/颜色） |
| POST | `/api/scan` | 触发目录扫描 |
| GET | `/api/scan/result` | 获取扫描结果（新文件列表） |
| POST | `/api/scan/import` | 批量导入扫描结果中的文件 |
| GET | `/api/stats` | 仪表盘统计数据 |
| GET | `/api/checklists` | 获取所有检查清单 |
| POST | `/api/checklists` | 创建检查清单 |
| GET | `/api/checklists/{id}` | 获取清单详情（含明细项） |
| PUT | `/api/checklists/{id}` | 修改检查清单 |
| DELETE | `/api/checklists/{id}` | 删除检查清单 |
| POST | `/api/checklists/{id}/items` | 添加检查项 |
| PUT | `/api/checklists/{id}/items/{item_id}` | 修改检查项 |
| DELETE | `/api/checklists/{id}/items/{item_id}` | 删除检查项 |
| POST | `/api/checklists/{id}/check` | 执行检查（参数：项目标签名），返回逐项结果 |
| POST | `/api/validate-filename` | 校验文件名是否符合命名模板（参数：filename, tag_id） |

## 八、非功能需求

- **性能**：单次上传支持最大 500MB 文件，批量导入 1000 个文件应在 30 秒内完成
- **安全**：仅本地访问（bind 127.0.0.1），不做用户认证
- **可靠性**：文件操作使用事务，入库失败不产生孤儿文件
- **可维护**：代码清晰分层（router → service → model），注释适度
